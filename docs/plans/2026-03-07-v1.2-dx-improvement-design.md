## 機能1: ゲートチェック統合

### 課題

フェーズ遷移に3ステップ必要:
```
oops gate red-to-green   ← 手動
oops phase green         ← 手動（しかもゲート失敗時は無駄）
```

### 設計

`oops phase <target>` 実行時にゲートチェックを自動実行する。

- `oops phase green` → red-to-green ゲート自動実行 → パスしたら遷移
- `oops phase refactor` → green-to-refactor ゲート自動実行 → パスしたら遷移
- ゲート失敗時はエラー表示して遷移しない
- `--skip-gate` フラグで従来通りゲートスキップ可能
- `--force` フラグは既存のまま（無効な遷移も強制）

### 修正ファイル

- `src/commands/phase.ts` — `setPhase()` 内でゲートチェックを統合
- `src/__tests__/phase-manager.test.ts` — ゲート統合のテスト追加

### 後方互換性

- `autoGateCheck` 設定フラグは引き続き尊重（false なら従来動作）
- `oops gate` コマンドは引き続き単独利用可能

## 機能2: denyガイダンス強化

### 課題

denyメッセージが事実のみで、次のアクションが不明:
```
🚫 Phase: RED - Only test files allowed. Cannot modify: src/index.ts
```

### 設計

フェーズ別の具体的な次ステップを表示:

**RED phase で実装ファイルをdeny:**
```
🚫 Oops #3: RED phase - Cannot modify implementation file
   File: src/index.ts

   現在: RED phase（テストファイルのみ編集可）
   次のステップ:
     1. テストファイルを作成/編集して失敗するテストを書く
     2. テストが失敗することを確認
     3. oops phase green でGREENフェーズへ移行
```

**GREEN phase でテストファイルをdeny:**
```
🚫 Oops #4: GREEN phase - Cannot modify test file
   File: src/__tests__/foo.test.ts

   現在: GREEN phase（実装ファイルのみ編集可）
   次のステップ:
     1. テストが通る最小限の実装を書く
     2. テストが全て通ることを確認
     3. oops phase refactor でREFACTORフェーズへ移行
```

### 修正ファイル

- `src/hooks/pre-tool-use.ts` — deny時のメッセージ拡充（RED/GREEN各ケース）

### 後方互換性

- denyの判定ロジックは変更なし
- メッセージが長くなるだけ

## 機能3: planコマンド簡素化

### 課題

```bash
oops plan create --goal "認証機能" --subtask "ログインAPI: desc" --subtask "ミドルウェア: desc"
oops plan add --subtask "name: description"
```
`--subtask` の繰り返しが冗長。

### 設計

**A. `oops plan create` の `--subtask` をオプショナルに**
- サブタスクなしで `oops plan create --goal "認証機能"` → 空プラン作成
- 後から `oops plan add` で追加

**B. `oops plan add` の引数簡素化**
- 変更前: `oops plan add --subtask "name: description"`
- 変更後: `oops plan add "name: description"` （位置引数化）
- `--subtask` フラグも引き続き使用可能（後方互換）

### 修正ファイル

- `src/cli/index.ts` — `plan add` コマンドの引数定義変更
- `src/commands/plan.ts` — `addSubtaskCommand()` の引数処理
- `src/__tests__/plan-manager.test.ts` — テスト追加

### 後方互換性

- `--subtask` フラグは引き続き動作
- 既存のプランファイルフォーマットは変更なし

## 機能4: init テストコマンド自動検出

### 課題

- テストコマンドが `npm test` 固定
- Python/Go/Rust プロジェクトでは手動設定が必要

### 設計

`oops init` 実行時にプロジェクトのファイルからテストコマンドを自動検出:

| 検出ファイル | テストコマンド |
|---|---|
| `package.json` | `npm test` |
| `pyproject.toml` or `pytest.ini` | `pytest` |
| `Cargo.toml` | `cargo test` |
| `go.mod` | `go test ./...` |
| `Makefile` (test ターゲット) | `make test` |
| 検出不能 | `npm test` (デフォルト) + 警告 |

**`oops init --test-command "..."` オプション追加:**
自動検出を上書きしたい場合に使用。

### 修正ファイル

- `src/commands/init.ts` — `detectTestCommand()` 関数追加、`initOops()` に統合
- `src/__tests__/init.test.ts` — 新規テスト

### 後方互換性

- 既存の `oops init` は package.json があれば `npm test` で変更なし
- `--test-command` なしの場合のみ自動検出が発動
